# 项目学习文档（Vue + FastAPI 记账应用 + AI）

本文档用于帮助你快速理解该项目的整体结构、关键模块的实现方式、以及 AI 功能的工作流（查询 + 写入提案 + 二次确认 + 记忆）。

## 1. 整体架构

- 前端：Vue 3 + Vite + Pinia + Vue Router
  - 负责登录、记账 CRUD、分类管理、统计展示、AI 对话 UI（含提案确认/取消/撤销）
- 后端：FastAPI + SQLAlchemy + SQLite（可切换 Postgres）
  - 提供 REST API（auth/users/accounts/categories/transactions/stats/ai）
  - AI 模块支持 rule 模式（本地规则兜底）与 LLM 模式（DeepSeek/OpenAI + LangChain 工具调用）

前后端通信方式：前端通过 `src/api/client.ts` 调用后端 `http://127.0.0.1:8000/api/...` 接口（本地开发默认）。

## 3. 核心业务实现方式

### 3.1 认证与用户体系（JWT）

- 注册：前端 `api.auth.register` 调 `POST /api/auth/register`；后端创建 user 并返回 user 信息
- 登录：前端 `api.auth.login` 调 `POST /api/auth/login`，拿到 `access_token` 存 localStorage
- 自动登录：路由守卫会在有 token 时 `auth.bootstrap()`，调用 `GET /api/users/me`
- 后端鉴权：每个需要登录的接口使用 `Depends(get_current_user)` 注入当前用户

## 4. AI 模块：工作流与实现方式

AI 是该项目的“增强模块”，它的核心目标是：
- 让用户用自然语言查询/统计
- 支持自然语言写入（新增流水、修改流水归类、创建分类等）
- 所有写入必须“二次确认”
- 有审计日志，并支持撤销最近一次 AI 操作
- 有会话记忆（conversation_id）

### 4.1 AI 对外接口

- 状态检查：`GET /api/ai/status`
  - 返回 AI_MODE、provider、model、langchain 是否安装、key 是否配置等
- 对话：`POST /api/ai/chat`
  - 入参：`message` + `conversation_id?`
  - 出参：`reply` + `cards` + `proposed_actions` + `conversation_id`
- 二次确认执行：`POST /api/ai/confirm`
  - 入参：`action_id`
  - 执行并写入审计日志（返回 audit_id/entity_id）
- 取消提案：`POST /api/ai/cancel`
- 撤销最近一次：`POST /api/ai/undo-last`

### 4.2 AI 的两条“理解路径”：LLM 优先 + 规则兜底

核心调度在 [ai/service.py](file:///d:/PROJECTS/note/api/app/ai/service.py)：

- 当 `AI_MODE=deepseek/openai`：
  - 优先调用 `run_langchain_chat(...)`（大模型理解 + 工具调用）
  - 如果大模型返回“未能完成工具调用”且没有 cards/actions，再回退到规则兜底逻辑（避免完全无结果）
- 当 `AI_MODE=rule`：
  - 不走大模型，只走规则理解（本地解析 + 直接产出提案/卡片）

这里的“规则兜底”不是为了替代大模型，而是为了：
- 对“高确定性输入”给出稳定结果（例如含金额、交易ID）
- 当大模型 tool call 失败时不至于直接断档

### 4.3 工具调用（Tool Calling）实现

大模型并不会直接访问数据库，它必须通过“工具”来做事：

- 查询工具（AI tools）
  - `get_stats_summary / get_stats_by_category / list_transactions`
  - 实际落到 [ai/tools.py](file:///d:/PROJECTS/note/api/app/ai/tools.py) 的 SQLAlchemy 查询
- 写入工具（只生成提案，不直接执行）
  - `propose_transaction_create / propose_transaction_update / propose_transaction_delete`
  - `propose_category_create / propose_category_create_and_assign`
  - 这些工具内部会创建 `AIPendingAction`

LangChain 的核心实现：
- [ai/langchain_agent.py](file:///d:/PROJECTS/note/api/app/ai/langchain_agent.py)
  - 系统提示：约束模型必须通过工具来访问数据，写操作必须走 propose_*
  - 绑定工具：`llm.bind_tools([...])`
  - 执行循环：获取 `tool_calls`，解析工具名/参数，执行并把 ToolMessage 回填给模型

### 4.4 “二次确认”与审计日志

写入动作不会直接落库，而是走：

1) 生成提案：创建一条 `AIPendingAction`（status=pending）
2) 用户确认：`/api/ai/confirm` 调用 `confirm_action`
3) 执行落库：真正创建/更新/软删除 Transaction 或创建 Category
4) 写审计：记录 `AuditLog`（before_json/after_json）

核心实现：
- [ai/actions.py](file:///d:/PROJECTS/note/api/app/ai/actions.py)
  - `create_pending_action`：生成提案
  - `confirm_action`：执行提案 + 写审计
  - `cancel_action`：取消提案
  - `undo_last_ai_action`：撤销最近一次 AI 操作（最小实现）

为什么要这样设计：
- 防止“模型误操作”导致数据被直接改坏
- 审计可追溯（谁、什么时候、改了什么）

### 4.5 AI 记忆（conversation_id）

为了让 AI 能“记住上一轮说过的话”，项目实现了简单的会话记忆：

- 前端在 [AIChatPage.vue](file:///d:/PROJECTS/note/src/pages/AIChatPage.vue) 保存 `conversationId`
  - 每次调用 `api.ai.chat` 都带上它
  - 新对话按钮会清空 conversationId
- 后端保存会话消息与状态
  - `AIConversation`：`state_json` 保存状态（draft、last_action_id 等）
  - `AIConversationMessage`：保存对话消息（user/assistant）
- 大模型模式下，后端会把最近 N 条消息作为 history 传入 `run_langchain_chat`

### 4.6 自然语言解析（规则兜底）

规则兜底主要集中在：
- [ai/nlp.py](file:///d:/PROJECTS/note/api/app/ai/nlp.py)：金额、时间、商户、交易ID、分类ID/分类名、归类目标等解析
- [ai/rule_agent.py](file:///d:/PROJECTS/note/api/app/ai/rule_agent.py)：rule 模式下的简化对话逻辑（查询/简单写入提案）
- [ai/service.py](file:///d:/PROJECTS/note/api/app/ai/service.py)：把 nlp 的结果组合成提案或卡片

典型例子：
- `我今天下午四点在肯德基花了50元` -> 解析金额/时间/商户 -> 生成 `transaction_create` 提案
- `帮我把那笔肯德基的消费归类到餐饮里` -> 解析分类名 + 搜交易 -> 生成 `transaction_update` 提案

## 6. 可扩展点（你继续开发时常用）

- 新增一种 AI “写操作”
  - 1) 在 `ai/actions.py` 增加 `kind` 分支（confirm_action 执行逻辑 + 审计）
  - 2) 在 `ai/langchain_agent.py` 增加 propose_ 工具（只生成 pending action）
  - 3) 在 `ai/service.py` 增加规则兜底（可选）
  - 4) 前端 AI 页会自动显示 proposed_actions（只要返回里有）
- 提升自然语言理解
  - 优先改 system prompt（约束模型怎么调用工具、如何补全缺省字段）
  - 再补 nlp 规则兜底（解决高频口语）
- 生产化建议
  - 把 sqlite 换成 Postgres
  - 引入 Alembic 做数据库迁移（避免字段变更只能靠 “ensure_sqlite_schema”）
